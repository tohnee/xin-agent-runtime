01

Vercel 发出的平台信号

这次判断大体成立，而且比"Vercel 又发了一个 agent 框架"更重要。原因不是 eve 本身有多新， 而是 Vercel 在 2026 年 6 月把一组过去分散在不同项目里的 runtime primitive 明确组合成了平台默认能力。

核心转变：框架是入口，运行时才是护城河。Vercel 这次释放的是从 「agent 能不能做」转向「agent 能不能稳定、安全、可追责地做」的产品分水岭信号。

六个关键支柱

PILLAR 01

Durability — 可恢复执行

Workflows 可暂停数分钟到数月，通过 deterministic replay 从精确位置恢复。WorkflowAgent 把 durable agent loop 直接带进 AI SDK 7。

PILLAR 02

Sandbox — 隔离执行环境

Firecracker microVM 语义，支持文件、端口、snapshot、drive 与 network policy，默认可持久化快照。

PILLAR 03

AI Gateway — 模型层解耦

统一 API、预算控制、监控、负载均衡和 fallback。将模型提供方从 agent 逻辑中完全解耦。

PILLAR 04

Connect — 短期凭证体系

把长期 token 改成按任务发放的短期 scoped credential，动作归因到具体用户。HMAC 签名防伪造。

PILLAR 05

Observability — 原生可观测

Dashboard 原生 logs、traces、spans，Drains 导出到外部系统，AI Gateway usage/spend 统一监控。

PILLAR 06

Evals — 评估一等公民

eve 框架把 evals 直接放进范式，支持按部署和定时运行。评估不是插件，是 CI 一部分。

重要背景：这个方向不是 Vercel 发明的。AgentScope 2.0（2026 年 5 月）和 DeerFlow 2.0（2026 年初）都已在往 runtime 化走。 Vercel 的差异在于：第一次由主流应用平台把这些能力产品化并默认化。

AgentScope 的价值不在"更多 feature"，而在于可塑性： 你可以自己决定 storage backend、message bus、IAM、tenant 隔离策略、observability 接入点。 这是把 runtime 视为工程资产而非外购服务的底层逻辑。对想做长期平台化演进、且有 Python 后台体系的企业， AgentScope 通常是最自然的选择。

架构关键信号：AgentScope Runtime 从独立仓库合并进主框架，说明团队已经明确地把 runtime 能力 定位为框架的核心层，而不是可选附件。这和 Vercel 把 Workflows/Sandbox 定位为平台核心是同向的判断，只是实现路径不同。

08

最终结论

核心命题

把这三者放在一起看，最关键的结论不是「谁更强」，而是 谁在为你承担 production runtime 的默认成本。 这是选型的真正分水岭，而不是 benchmark 分数或 feature list 长度。

Vercel 这次真正证明的是：agent framework 的下一站确实不是再多一个 SDK，而是生产级运行时。 它以平台信号的方式将 durable execution、sandbox、approvals、connect、observability 打包为默认件， 这对整个行业的产品定义都有影响。

三条选型路径

想要最快落地、最少自建、最强平台默认件：选 Vercel。 你买到的不是「更智能的模型」，而是更短的生产路径—— durable orchestration、sandbox、approvals、connect、spend monitoring 这些底层件你一个都不用自组。 代价是平台边界强，且部分企业功能仍处 beta/private beta 阶段。

想要自控边界、长期平台化、可移植和可自托管：选 AgentScope。 它现在最像「企业自己掌控 runtime 边界」的底座：permission、workspace、service、middleware、tracing、 evaluation 都在持续补齐，且 runtime 能力已从独立仓库合并进主框架。 对 Python/Java 后台体系、私有化部署、重合规企业，AgentScope 的可塑性通常高于 Vercel。

面向 deep research / coding / 内容产物生成，需要「一个已经很能打的 super-agent 系统」：选 DeerFlow。 但在企业环境里，不要把 DeerFlow 当「开箱即用的治理平台」，而应当作「高能力 harness」—— 外面再补企业认证、策略授权、审计、网络隔离和评估体系。

一句话重新定义目标

未来真正有价值的 Agent 平台，不只是「可恢复、可审计、可审批、可验收、可复用的工作台」， 而是把这些能力做成默认 runtime，并且能与你的部署边界、身份体系、审计体系和成本治理体系对齐的工作台。
